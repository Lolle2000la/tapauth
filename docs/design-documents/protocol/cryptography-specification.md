# Cryptography Specification

This document specifies the exact cryptographic algorithms to be used in the TapAuth protocol. The chosen algorithms are current industry standards, selected for their high performance, strong security guarantees against all classical attacks, and wide availability in modern cryptographic libraries.

## 1. Asymmetric Cryptography

The protocol uses modern Elliptic Curve Cryptography (ECC) based on Curve25519, which provides a ~128-bit security level.

### 1.1. Key Exchange

* **Algorithm**: **X25519**
* **Purpose**: Used during the initial pairing to perform an Elliptic Curve Diffie-Hellman (ECDH) key exchange, generating a temporary shared secret.

### 1.2. Digital Signature Algorithm

* **Algorithm**: **Ed25519**
* **Purpose**: Used to sign all `AuthenticationRequest`, `AuthenticationGrant`, and other core messages for authenticity and integrity before they are encrypted.

## 2. Symmetric Cryptography

### 2.1. Authenticated Encryption with Associated Data (AEAD)

* **Algorithm**: **AES-256-GCM**
* **Purpose**: Provides authenticated encryption for all network communication. It is used with two different types of symmetric keys depending on the context.

### 2.2. Key Derivation Function (KDF)

* **Algorithm**: **HKDF-SHA256**
* **Purpose**: Used to derive cryptographically strong symmetric keys from the shared secret generated by the X25519 key exchange.

### 2.3. Symmetric Key Types

The protocol uses two distinct types of symmetric keys:

* **Pairing Symmetric Key (`PSK`)**:
    * **Scope**: Ephemeral, session-only.
    * **Derivation**: Derived from the X25519 ECDH exchange during the initial pairing process using HKDF-SHA256.
    * **Purpose**: Its sole function is to create a secure, temporary channel to encrypt the long-term `CSK` as it is transmitted from the Client to the Server.
    * **Lifecycle**: The `PSK` **must** be securely discarded by both the Client and Server immediately after the pairing process is successfully completed or fails.

* **Client Symmetric Key (`CSK`)**:
    * **Scope**: Long-term.
    * **Generation**: A cryptographically secure 32-byte key generated once by the Client application, typically upon first pairing.
    * **Purpose**: This is the primary key used to encrypt all post-pairing communication. It is shared by a single Client with all of its paired Servers, allowing the Client to broadcast a single encrypted message that can be decrypted by any authorized Server.
    * **Lifecycle**: The `CSK` is stored securely on the Client and on all paired Servers until a device is explicitly un-paired or the key is rotated.
